# Assistant 三栏改造方案（草案 v2，待审核）

> 目标：把 Assistant 从"一个网页标签 / 窄侧栏"改造成浏览器固定的原生部分。
> 布局：**左 = 任务切换栏 / 中 = AI 对话 / 右 = 浏览器**，右栏的标签是普通浏览器那种横在顶部的样式。
> 本文是分阶段实施方案，每阶段独立交付、独立可用。阶段拆得很细，前三个阶段都是扩展层小改（1-2 周一个），后面两个是 Chromium 大改。

---

## 一、最终目标长什么样（先看结果）

```
┌──────────┬─────────────┬──────────────────────────────┐
│  任务栏   │   对话栏     │  浏览器栏                     │
│          │             │  [网页A] [网页B] [网页C]    +  │  ← 普通浏览器
│  ▸ 任务1 │             │──────────────────────────────│     式顶部标签
│  ▸ 任务2 │  你: 帮我.. │                              │
│  ▸ 任务3 │  AI: 好的.. │                              │
│          │             │                              │
│  + 新任务 │  [输入框]   │       当前网页内容            │
│          │       [发送] │                              │
│          │             │                              │
└──────────┴─────────────┴──────────────────────────────┘
    ~15%        ~20%                 ~65%
   固定面板      固定面板            固定面板（标签在顶部）
```

三个区域**都是浏览器原生面板，互不覆盖**：
- 切换网页标签，左侧任务栏和中间对话栏**不消失、不重载**
- 切换任务，右侧整个浏览器（包括顶部标签栏）换成那个任务对应的网页集合
- 中间对话栏跟着任务走，任务和对话一一对应
- 一个任务可以对应**多个网页标签**（普通浏览器那样的多标签）

这就是"Assistant 固定，不再是一个网页标签"的含义。

## 二、当前问题（阶段 0：现状）

现在 Assistant 有两个突出问题：

### 问题 1：右上角 Assistant 按钮点不开

**现象**：点浏览器工具栏上的 Assistant 按钮（扩展 action 图标）没反应，每次要进 `chrome://extensions` 才能打开。

**疑似根因**（有代码依据）：
- [toggleSidePanel.ts:38-40](packages/browseros-agent/apps/app/lib/browseros/toggleSidePanel.ts#L38-L40)：当 `perWindow=false` 时，初始化执行 `chrome.sidePanel.setOptions({enabled: false})`——**把全局 sidePanel 禁用了**
- [toggleSidePanel.ts:51](packages/browseros-agent/apps/app/lib/browseros/toggleSidePanel.ts#L51)：默认 fallback 也是 `false`
- 按钮点击链路：[background/index.ts:61-64](packages/browseros-agent/apps/app/entrypoints/background/index.ts#L61-L64) → `toggleSidePanel` → per-tab 分支 → `chrome.sidePanel.browserosToggle`，在全局 `enabled:false` 状态下打不开

**关键**：这个 bug 和"窗口不固定"是**同一个根因**。把默认改成 `perWindow=true`，初始化会执行 `setOptions({enabled: true, path})`——按钮通了，窗口也固定了，一次解决。

### 问题 2：Assistant 不固定 + 看不到 AI 开的网页

- **形态 A**：Assistant 是新标签页（[wxt.config.ts:39-41](packages/browseros-agent/apps/app/wxt.config.ts#L39-L41) `newtab = app.html`），切标签整个消失
- **形态 B**：sidepanel 默认 per-tab，切标签跟着换/消失
- AI 开的网页和用户自己开的网页混在同一个标签条，分不清谁是谁
- 没有"多任务"概念，开多个任务会互相串

## 三、分阶段方案

总览（每阶段独立交付价值，不用等后面阶段做完才有用）：

| 阶段 | 做什么 | Assistant 固定性 | 工作量 |
|---|---|---|---|
| **阶段 1** | 固定窗口 + 修按钮打不开 | 窗口级固定 | **~1 周** |
| 阶段 2 | 任务归组接通（AI 开的网页自动归组） | 窗口级固定 | ~1 周 |
| 阶段 3 | 任务列表 UI（多任务切换、可见） | 窗口级固定 | 1-2 周 |
| 阶段 4 | 原生三栏（Chromium 改造）★ 目标态 | 浏览器级固定 | 1-2 个月 |
| 阶段 5 | Workspace 多任务完全隔离 | 浏览器级固定 + 物理隔离 | 2-3 个月 |

前三个阶段是扩展层小步快跑，每个 1-2 周就能交付一个可见改善。阶段 4、5 是 Chromium 大改。

---

### 阶段 1：固定窗口 + 修复按钮打不开（~1 周）★ 第一步

**目标**：用最小改动，让"点工具栏按钮能打开 Assistant"和"切网页标签 Assistant 不消失"这两件最基本的事先成立。**只做这一件事**，不碰任务化、不碰归组、不碰 UI 重构。

**界面**（改完后的样子）：
```
┌──────────────────────────────────────┬──────────────┐
│ [网页1] [网页2] [网页3]      + 新建   │              │
│  (普通浏览器顶部标签，不变)            │              │
├──────────────────────────────────────┤   Assistant   │
│                                      │   侧边栏      │
│                                      │  （现有 UI，  │
│                                      │   不重构）    │
│         网页内容（普通浏览器）          │              │
│                                      │  对话/历史    │
│                                      │  （维持现状） │
│                                      │              │
└──────────────────────────────────────┴──────────────┘
  浏览器主区域（不变）                右侧窗口固定侧边栏
```

**这个阶段完成的功能**（就两件）：
1. **点工具栏 Assistant 按钮能打开侧边栏**（修复问题 1）
2. **切换网页标签，侧边栏不消失、不重载**（窗口级固定）

**改动范围**（极小，3 处）：
- [toggleSidePanel.ts](packages/browseros-agent/apps/app/lib/browseros/toggleSidePanel.ts)：`sidePanelPerWindowStorage` 的默认值从 `false` 改成 `true`。这样初始化走 `setOptions({enabled: true, path})` 分支，按钮通了
- [toggleSidePanel.ts:51](packages/browseros-agent/apps/app/lib/browseros/toggleSidePanel.ts#L51)：fallback 也改成 `true`
- [wxt.config.ts:39-41](packages/browseros-agent/apps/app/wxt.config.ts#L39-L41)：去掉 `chrome_url_overrides.newtab = app.html`，Assistant 不再是新标签页（避免"切走就消失"的形态 A）。

**验收标准**：
- 点工具栏 Assistant 按钮，侧边栏打开
- 打开多个网页标签来回切，侧边栏一直在、不重载、对话内容不丢
- 打开新标签页不再显示 Assistant 全屏 UI（回到普通新标签页）

**这个阶段不做**：任务列表、任务归组、AI 开的网页可视化、UI 重构——都留给后面阶段。

---

### 阶段 2：任务归组接通（~1 周）

**目标**：让 AI 在执行任务时开的每一个网页，自动归到带颜色和任务名的标签组。解决"AI 开的网页和用户的混在一起分不清"。

**界面**：
```
┌──────────────────────────────────────┬──────────────┐
│ [网页1] [🔵 任务A · p4 p5] [网页2]  + │              │
│  (AI 开的网页自动归组，带颜色标记)     │              │
├──────────────────────────────────────┤   Assistant   │
│                                      │   侧边栏      │
│         网页内容（普通浏览器）          │ （现有 UI）   │
│                                      │              │
└──────────────────────────────────────┴──────────────┘
```

**这个阶段完成的功能**：
1. 每次新建一个对话任务，自动创建一个 Chrome 标签组（有颜色、有任务名）
2. AI 在这个任务里开的每一个网页，自动归入这个组
3. 顶部标签条上能看到"哪些是 AI 为这个任务开的"

**改动范围**：
- [tool-adapter.ts](packages/browseros-agent/apps/server/src/agent/tool-adapter.ts)：`BrowserToolSetOptions` 补上 `defaultTabGroupId` 和 `defaultWindowId` 字段——**这是当前内置路径的关键缺口**，AI 开的网页现在不归组就是因为这里没传
- [ai-sdk-agent.ts](packages/browseros-agent/apps/server/src/agent/ai-sdk-agent.ts)：把 `defaultTabGroupId` 透传到 `buildBrowserToolSet`
- [chat-service.ts](packages/browseros-agent/apps/server/src/api/services/chat-service.ts)：新建会话时调 `Browser.createTabGroup` 建组，把组 ID 存进 session，传给 agent

**验收标准**：
- 新开一个任务，让 AI 打开 3 个网页，这 3 个网页在顶部标签条上聚成一个彩色组
- 开两个任务，两个任务的网页各自归组，颜色不同，不混

**这个阶段不做**：任务列表 UI、任务切换、tab 树可视化——留给阶段 3。

---

### 阶段 3：任务列表 UI（1-2 周）

**目标**：在侧边栏里显示任务列表 + 每个任务的网页树，让用户能看见和管理多个并行任务。

**界面**：
```
┌──────────────────────────────────────┬──────────────┐
│ [网页1] [🔵任务A] [网页2]    + 新建   │  任务列表     │
├──────────────────────────────────────┤              │
│                                      │ ▸ 任务A (3页) │
│                                      │ ▸ 任务B (1页) │
│         网页内容（普通浏览器）          │ ▸ 任务C (5页) │
│                                      │ ──────────   │
│                                      │ 对话(当前任务)│
│                                      │ 你: ...      │
│                                      │ AI: ...      │
└──────────────────────────────────────┴──────────────┘
```

**这个阶段完成的功能**：
1. 侧边栏上半部分显示所有任务，每个任务显示它有几个网页
2. 点任务可展开，看到 AI 为它开的所有网页列表，点网页跳过去
3. 点任务切换：对话切到那个任务，浏览器跳到那个任务的标签组
4. 同时开多个任务并行，互不干扰

**改动范围**：
- 侧边栏 UI 重构：上半任务列表 + 每 task 的网页树，下半对话
- 调用 `tab_groups list` + `tabs list` 拉数据（工具已就绪）
- 任务切换的联动逻辑
- Chromium 小 patch：放宽侧边栏宽度上限（项目已有 `browserOS` 自定义权限和 sidePanel patch 先例，通道是通的）

**验收标准**：
- 同时开 3 个任务，侧边栏能看到全部 3 个任务及其网页数
- 点任务栏的任务，对话和浏览器都跟着切
- 展开任务能看到 AI 开的每个网页，点一下跳过去

**做完阶段 3 后还差什么**（驱动阶段 4）：
- 侧边栏还是窄，任务列表和对话挤在一起
- 浏览器主区域和侧边栏是"挤"的关系，不是平等的三栏
- 还是扩展 UI 层，不是原生浏览器 chrome

### 阶段 4：原生三栏（Chromium 改造，1-2 个月）★ 目标态

**目标**：把第一节那张目标图做出来。三个区域都是浏览器原生面板，Assistant 真正成为浏览器的固定部分。

**界面**（就是第一张目标图）：
```
┌──────────┬─────────────┬──────────────────────────────┐
│  任务栏   │   对话栏     │  [网页A] [网页B] [网页C]    +  │
│          │             │──────────────────────────────│
│  ▸ 任务1 │  你: ...    │                              │
│  ▸ 任务2 │  AI: ...    │                              │
│  ▸ 任务3 │             │       网页内容                │
│          │             │                              │
│  + 新任务 │  [发送]     │                              │
└──────────┴─────────────┴──────────────────────────────┘
   原生面板     原生面板        原生面板（标签在顶部）
```

**这个阶段完成的功能**（在阶段 3 基础上新增）：
1. **原生三栏布局**：左/中/右三栏都是浏览器自己的面板（像 IDE 那样），可拖拽调宽、可折叠
2. **右栏是完整浏览器**：顶部是普通横向标签条（用户熟悉的样式），下面是网页内容
3. **标签条按任务过滤**：右栏顶部只显示**当前选中任务**的网页标签，切换任务 = 整个标签条换成那个任务的网页
4. **对话栏常驻**：边看网页边和 AI 对话，两边同时可见
5. **Assistant 不再是标签**：彻底去掉"Assistant 作为新标签页"的形态

**改动范围**：
- Chromium patch（`chrome/browser/ui/views/`）：原生三栏可拖拽面板
- 右栏内嵌真实网页内容（用原生 webContents，不是 iframe，突破网页嵌入限制）
- 任务和标签的绑定从"Chrome 标签组"升级为原生面板 context
- 对话栏做成原生面板，支持折叠成抽屉（让浏览器偶尔可以撑到 ~85% 宽）

**验收标准**：
- 切换网页标签，任务栏和对话栏纹丝不动
- 切换任务，右栏的标签条和网页内容整个换掉
- 三个面板都能拖拽调宽
- 关掉对话栏抽屉后，浏览器可以占大部分宽度

**可行性依据**：项目 manifest 里已经有 `browserOS` 自定义权限和 `chrome.sidePanel.open({windowId})` 这种非标准 API（[wxt.config.ts:65](packages/browseros-agent/apps/app/wxt.config.ts#L65)），说明 **BrowserOS fork 已经 patch 过 sidePanel**，改浏览器 UI 层的通道已经打通，阶段 4 不是从零开始。

### 阶段 5：Workspace 多任务完全隔离（2-3 个月）

**目标**：每个任务是一个独立 Workspace，真正多任务并行互不干扰。

**界面**（和阶段 4 类似，但任务栏换成 Workspace 切换器）：
```
┌──────────┬─────────────┬──────────────────────────────┐
│ Workspace│   对话栏     │  [网页A] [网页B]           +  │
│  切换器   │             │──────────────────────────────│
│ ■ 工作 A │  你: ...    │                              │
│ □ 工作 B │  AI: ...    │                              │
│ □ 个人   │             │       网页内容                │
│ □ 研究   │             │                              │
│ + 新建WS │             │                              │
└──────────┴─────────────┴──────────────────────────────┘
  切换 Workspace = 整窗换景（对话+网页整体换）
```

**这个阶段完成的功能**：
1. **任务 = 独立 Workspace**：每个任务有自己的 cookie、登录态、历史、标签集合，任务间完全隔离（类似 Arc Space、Edge Workspaces）
2. **后台任务继续跑**：AI 在非活跃 Workspace 里可以继续工作，用户在当前 Workspace 不受打扰
3. **多账号场景**：不同任务登不同账号，不串
4. **可选增强**：Workspace 之间拖拽网页、任务模板、子任务

**改动范围**：
- Workspace 模型（基于 Chromium profile 或 CDP target partition）
- 跨 Workspace 调度和后台运行
- 任务栏 UI 升级为 Workspace 切换器

**验收标准**：
- 两个任务登同一个网站的不同账号，互不干扰
- 任务 A 在后台跑 AI，用户在任务 B 浏览，B 看不到 A 的网页
- 切换 Workspace 是瞬间整体换景
